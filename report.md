# Лабораторная работа №8
## Разработка на основе структурированных промптов

### Цель работы

Изучить и применить методологию разработки на основе структурированных промптов (Structured-Prompt-Driven Development, SPDD) для управляемой генерации программного кода с помощью большой языковой модели. Научиться использовать структурированные промпты как версионируемые, проверяемые и повторно используемые артефакты разработки.

Методологической основой является статья Martin Fowler / Thoughtworks «Structured-Prompt-Driven Development»: https://martinfowler.com/articles/structured-prompt-driven/

В статье SPDD рассматривается как подход, в котором промпты становятся полноценными артефактами разработки: они хранятся в системе контроля версий, проходят проверку, уточняются вместе с кодом и задают границы генерации. Для структурирования промпта используется REASONS Canvas: Requirements, Entities, Approach, Structure, Operations, Norms, Safeguards.

### 1. Выбор задачи для разработки

В рамках лабораторной работы была выбрана задача разработки backend-части курсового проекта TextGuard AI на основе структурированных промптов.

Тема курсового проекта: B2B-сервис автоматической AI-модерации и детоксикации русскоязычного текстового контента.

Важно ограничить область работы. В рамках данной лабораторной работы рассматривалась только backend-часть приложения. Frontend, обучение моделей, подготовка датасетов, подбор архитектуры RuBERT/ruT5 и изменение локальных весов не разрабатывались мной и не входят в область данного SPDD-задания.

Backend разрабатывался как серверный слой поверх уже существующих ML-сервисов:

- прием текстового сообщения через REST API;
- авторизация внешнего клиента по API-ключу;
- запуск классификации токсичности через `ClassifierService`;
- запуск детоксикации через `DetoxService` только при токсичности;
- возврат JSON-ответа с результатом анализа;
- учет запросов по workspace и API-ключу;
- управление тарифами Free, Pro и Business;
- отображение статистики через dashboard;
- сохранение legacy-совместимости со старым endpoint.

Основные файлы backend:

- `backend/main.py`;
- `backend/deps.py`;
- `backend/auth.py`;
- `backend/store.py`;
- `backend/plans.py`;
- `backend/user_store.py`;
- `backend/routes/v1_moderation.py`;
- `backend/routes/moderation.py`;
- `backend/routes/api_keys.py`;
- `backend/routes/subscriptions.py`;
- `backend/routes/dashboard.py`;
- `backend/routes/auth.py`;
- `backend/routes/keys.py`;
- `backend/services/classifier_service.py`;
- `backend/services/detox_service.py`.

### 2. Изначальные требования к модулю

Перед составлением промптов были сформулированы требования к backend-модулю.

Функциональные требования:

1. backend должен запускаться как FastAPI-приложение;
2. приложение должно подключать CORS для локального frontend;
3. должен существовать новый B2B endpoint `POST /api/v1/moderation/analyze`;
4. B2B endpoint должен использовать `Authorization: Bearer <API_KEY>`;
5. должен сохраняться legacy endpoint `POST /api/moderate` с `X-API-Key`;
6. backend должен вызывать `ClassifierService.predict()` для расчета токсичности;
7. backend должен вызывать `DetoxService.detoxify()` только при токсичном тексте;
8. ответ нового endpoint должен быть в camelCase: `originalText`, `cleanedText`, `status`, `isToxic`, `scores`;
9. должны быть endpoint-ы управления API-ключами;
10. должны быть endpoint-ы списка и выбора тарифов;
11. должен быть endpoint dashboard со статистикой использования;
12. должен быть admin endpoint для служебной выдачи ключей;
13. должна быть mock-аутентификация для кабинета;
14. должны обрабатываться ошибки пустого текста, отсутствующего ключа, неверного ключа и превышения лимита.

Нефункциональные требования:

- не изменять локальные веса моделей;
- не переносить ML-логику в роутеры;
- не использовать внешние облачные сервисы во время инференса;
- использовать русскоязычные сообщения ошибок;
- не раскрывать полный API-ключ в списках;
- хранить промпты как отдельные Markdown-файлы;
- явно указать, что in-memory store является демонстрационным ограничением.

### 3. Анализ требований и контекста существующей кодовой базы

Перед составлением структурированного промпта был изучен контекст проекта. Кодовая база уже содержала ML-сервисы `classifier_service.py` и `detox_service.py`, поэтому backend должен был не обучать модели заново, а предоставить стабильный API-контракт для их использования.

Были выделены доменные концепции:

- `текстовое сообщение` - входная строка, которую отправляет клиент;
- `API-ключ` - средство доступа внешнего клиента к B2B API;
- `workspace` - демонстрационное пространство клиента;
- `тариф` - набор лимитов запросов и количества ключей;
- `scores` - вероятности классов токсичности;
- `isToxic` - итоговый флаг токсичности;
- `cleanedText` - результат детоксикации токсичного сообщения;
- `RequestLog` - краткая запись о запросе для dashboard;
- `legacy endpoint` - старый маршрут `/api/moderate`, который нельзя ломать;
- `admin-token` - служебный токен для административной выдачи ключей.

Бизнес-правила backend:

1. запрос без валидного ключа не должен запускать ML-инференс;
2. пустой текст должен отклоняться до обращения к модели;
3. безопасный текст не должен отправляться в `DetoxService`;
4. токсичный текст должен получить `cleanedText`;
5. успешный запрос должен увеличивать `requests_count`;
6. dashboard должен показывать использование ключа и недавние запросы;
7. список ключей должен отдавать preview, а не полный ключ;
8. legacy endpoint должен сохранять snake_case-формат ответа;
9. тариф должен ограничивать число запросов и ключей.

Технические риски:

1. тяжелые модели нельзя загружать на каждый запрос;
2. frontend ожидает camelCase-поля в новом API;
3. старые клиенты могут использовать `/api/moderate`;
4. без проверки лимита можно перегрузить локальный сервис;
5. in-memory store очищается после перезапуска;
6. ключ нельзя полностью показывать в списке;
7. session token для кабинета нельзя смешивать с API key для B2B-анализа.

Стратегическое решение:

Разработка должна быть выполнена в `backend/` без изменения frontend и без переобучения моделей. Взаимодействие строится через следующие endpoint-ы:

- `POST /api/v1/moderation/analyze`;
- `POST /api/moderate`;
- `GET /api/subscriptions/plans`;
- `POST /api/subscriptions/select`;
- `GET /api/api-keys`;
- `POST /api/api-keys/generate`;
- `DELETE /api/api-keys/{key}`;
- `GET /api/dashboard`;
- `POST /api/auth/login`;
- `GET /api/auth/me`;
- `POST /api/auth/logout`;
- `POST /api/admin/keys`;
- `GET /api/admin/keys`;
- `DELETE /api/admin/keys/{key}`.

### 4. Структурированный промпт по REASONS Canvas

На основе анализа была составлена цепочка структурированных промптов по схеме REASONS Canvas. Промпты использовались не как одноразовый запрос, а как последовательный план backend-разработки: от анализа контекста до финальной проверки контрактов.

В работе использовались следующие версии промптов:

- `prompts/prompt_v0.md` - анализ контекста backend и границ ответственности;
- `prompts/prompt_v1.md` - проектирование B2B API и маршрутов FastAPI;
- `prompts/prompt_v2.md` - проектирование API-ключей, тарифов и in-memory store;
- `prompts/prompt_v3.md` - реализация основного endpoint `/api/v1/moderation/analyze`;
- `prompts/prompt_v4.md` - реализация управления ключами, подписками и dashboard;
- `prompts/prompt_v5.md` - сохранение legacy `/api/moderate` и admin endpoint;
- `prompts/prompt_v6.md` - генерация backend-тестовых сценариев;
- `prompts/prompt_v7.md` - финальная проверка контрактов, ошибок и соответствия проекту.

Такая структура позволила представить backend-разработку как управляемый процесс. Каждый промпт решал отдельную задачу и ограничивал генерацию конкретной частью сервера. Это снизило риск того, что LLM начнет менять frontend, веса моделей или обучающие скрипты.

Использование REASONS Canvas позволило заранее зафиксировать:

- что именно должно быть реализовано;
- какие сущности участвуют в задаче;
- какой подход выбран;
- в каких файлах должны находиться изменения;
- какие операции нужно выполнить;
- какие инженерные нормы соблюдать;
- какие ограничения нельзя нарушать.

### 5. Генерация кода с помощью LLM

Для генерации и уточнения backend-кода использовалась LLM ChatGPT. Промпты были сохранены как отдельные файлы, чтобы их можно было проверять и версионировать вместе с результатом разработки.

Работа выполнялась итерационно:

1. `prompt_v0` - зафиксированы границы проекта: backend разрабатывается мной, frontend и ML-обучение не входят в задачу;
2. `prompt_v1` - спроектирована структура FastAPI-роутеров и список endpoint-ов;
3. `prompt_v2` - сформированы `store.py` и `plans.py` для workspace, ключей, логов и тарифов;
4. `prompt_v3` - добавлена основная логика B2B-анализа текста через `/api/v1/moderation/analyze`;
5. `prompt_v4` - добавлены маршруты управления ключами, подписками и dashboard;
6. `prompt_v5` - сохранена legacy-совместимость, admin endpoint и mock-auth;
7. `prompt_v6` - подготовлены тестовые сценарии для функциональной проверки backend;
8. `prompt_v7` - проведена финальная проверка контрактов, ошибок, лимитов и соответствия структуре проекта.

Таким образом, backend был не просто описан одним большим запросом, а последовательно уточнялся через серию структурированных промптов. Каждый следующий промпт опирался на результат предыдущего и сужал область генерации.

В результате backend представлен следующими файлами:

- `backend/main.py` - точка входа FastAPI и регистрация роутеров;
- `backend/deps.py` - Bearer-auth и admin-token;
- `backend/auth.py` - legacy X-API-Key;
- `backend/store.py` - workspace, ключи, счетчики и журнал;
- `backend/plans.py` - тарифы Free, Pro и Business;
- `backend/user_store.py` - mock-сессии пользователя;
- `backend/routes/v1_moderation.py` - основной B2B endpoint анализа;
- `backend/routes/api_keys.py` - операции с ключами;
- `backend/routes/subscriptions.py` - тарифы;
- `backend/routes/dashboard.py` - статистика кабинета;
- `backend/routes/auth.py` - mock-login;
- `backend/routes/keys.py` - admin-операции;
- `backend/routes/moderation.py` - legacy endpoint.

### 7. Функциональная проверка и корректировка промпта

После нескольких этапов генерации была проведена функциональная проверка. Проверка показала, что базовая структура backend, основной endpoint анализа и вызов ML-сервисов соответствуют задаче, однако потребовались уточнения:

- новый B2B endpoint должен использовать camelCase-ответ, а legacy endpoint - snake_case;
- API-ключи должны иметь preview в списке;
- при превышении лимита должен возвращаться HTTP 429;
- dashboard должен агрегировать usage по ключам workspace;
- mock-auth должен быть отделен от API-key авторизации;
- admin endpoint должен быть защищен отдельным `X-Admin-Token`;
- пустой текст должен отклоняться до запуска модели.

Для исправления этих проблем были созданы последующие промпты `prompt_v4.md`, `prompt_v5.md`, `prompt_v6.md` и `prompt_v7.md`. В соответствии с подходом SPDD уточнения фиксировались сначала в промптах, а затем переносились в код. Это позволило сохранить историю решений: почему появились отдельные роутеры, тарифы, dashboard, legacy-совместимость и финальная проверка контрактов.

### 8. Оценка соответствия итогового кода требованиям

Итоговый backend соответствует исходным требованиям.

Выполнено:

4. реализована Bearer-авторизация для B2B-запросов;
5. сохранен legacy endpoint `POST /api/moderate`;
6. реализованы API-ключи формата `textguard_sk_...`;
7. добавлены тарифы Free, Pro и Business;
8. реализовано ограничение числа запросов по тарифу;
9. реализовано ограничение числа ключей по тарифу;
10. добавлен dashboard со статистикой использования;
11. добавлены admin-операции с `X-Admin-Token`;

Оставшиеся ограничения:

- in-memory store очищается после перезапуска backend;
- настоящая регистрация пользователей и платежная система не реализованы;
- качество ML-ответа зависит от локальных весов моделей;
- автоматизированные API-тесты можно добавить как следующий этап, в лабораторной работе описаны функциональные сценарии.

### 9. Вывод

В ходе лабораторной работы была изучена и применена методология Structured-Prompt-Driven Development. На примере backend-части курсового проекта TextGuard AI были подготовлены требования, проведен анализ контекста, выделены доменные сущности, бизнес-правила, технические риски и стратегические ограничения.

Была составлена серия структурированных промптов по REASONS Canvas. Промпты использовались как самостоятельные артефакты разработки: от анализа backend-контекста и проектирования API до генерации отдельных частей серверной логики, тестовых сценариев и финальной проверки.

В результате была оформлена SPDD-документация для backend-модуля, подготовлены тестовые сценарии и выполнена оценка соответствия результата требованиям. Цель лабораторной работы достигнута: структурированные промпты были применены как управляемый, проверяемый и повторно используемый инструмент backend-разработки.
